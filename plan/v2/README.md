# v2 실행 계획

## 목표

`v2`의 목적은 `v1`의 단순 데모를 넘어서, gateway를 공통 정책 집행 지점으로 확장하는 것이다.

즉, `v2`부터는 "프록시가 된다"가 아니라 아래를 보여줘야 한다.

- 인증과 공통 정책이 서비스 바깥의 gateway 계층에서 동작한다
- 서비스별로 정책을 다르게 적용할 수 있다
- ingress gateway가 실제 정책 계층으로 작동하기 시작한다

## 범위

포함:

- 기본 인증 필터
- 공통 request ID / trace header 처리
- 요청/응답 메타데이터 표준화
- route별 정책 분기
- 표준 에러 응답
- 설정값 일부 외부화

제외:

- 중앙 control plane
- 동적 플러그인 배포
- 정교한 canary rollout
- 고급 quota 저장소

## 실행 단위

### 1. 인증 정책 추가

- JWT 또는 API key 중 하나를 선택해 기본 인증 구현
- 인증 성공/실패 처리 규칙 정의
- 인증 실패 시 표준 에러 응답 형식 적용

완료 기준:

- gateway 계층에서 인증이 수행되고, 인증 실패 요청은 upstream으로 가지 않는다

### 2. 공통 요청 메타데이터 표준화

- request ID 생성 또는 전달
- trace 관련 헤더 전달 규칙 정리
- 조직 공통 헤더 추가 규칙 정리

완료 기준:

- upstream은 gateway가 부여한 공통 메타데이터를 일관되게 받는다

### 3. 정책 분기 구조 도입

- route별로 서로 다른 정책을 적용할 수 있는 구조 추가
- `/users/*` 와 `/orders/*` 에 서로 다른 정책 조합을 줄 수 있게 구성

완료 기준:

- 최소 2개 이상의 route에 서로 다른 정책이 적용된다

### 4. WASM 내부 구조 정리

- 단일 거대 필터로 계속 가기보다 내부 모듈을 분리
- 필요하면 인증/관측성 중심으로 2~3개 필터 구조 검토
- 필터 책임과 순서 초안 정의

완료 기준:

- 정책 로직이 이후 분리 가능한 구조로 정리된다

### 5. 설정 외부화 시작

- 환경변수 또는 설정 파일 도입
- 인증 키, 정책 on/off, route별 옵션을 코드에서 분리

완료 기준:

- 주요 정책 설정값이 코드 하드코딩에서 벗어난다

### 6. 기본 운영 규칙 정리

- fail-open / fail-close 기본값 정의
- 인증 실패 로그 규칙 정의
- 정책 오류 발생 시 응답 원칙 정의

완료 기준:

- 장애나 정책 실패 시 gateway가 어떻게 행동하는지 명확해진다

### 7. 테스트 시나리오 확대

- 인증 성공/실패
- route별 정책 차이
- trace/request ID 전달
- 정책 오류 시 동작

완료 기준:

- 단순 라우팅 테스트를 넘어 정책 동작 검증이 가능하다

## 최종 완료 기준

- gateway가 공통 정책 계층으로 동작한다
- 서비스 코드 수정 없이 정책을 앞단에서 적용할 수 있다
- `v3`에서 플러그인 체인과 운영 구조를 얹을 수 있는 기반이 생긴다
