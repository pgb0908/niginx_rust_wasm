# v3 실행 계획

## 목표

`v3`의 목적은 gateway를 운영 가능한 플러그인 플랫폼 초안으로 끌어올리는 것이다.

이 단계부터는 단순 기능 추가보다 아래를 다뤄야 한다.

- 여러 정책 필터를 조합하는 구조
- 설정과 정책의 외부화
- 운영자가 배포, 변경, 롤백을 생각할 수 있는 구조

즉, `v3`는 "ingress gateway"에서 "플러그인 기반 gateway 플랫폼 초안"으로 넘어가는 단계다.

## 범위

포함:

- 다중 WASM 필터 체인
- 필터 순서/우선순위/중단 조건
- 정책 설정 외부화
- route/service/tenant 단위 정책 적용
- observability 연동 초안
- 롤백과 호환성 규칙 초안

제외:

- 완전한 control plane 제품화
- 대규모 멀티 리전 운영
- 완성형 관리자 UI

## 실행 단위

### 1. 다중 필터 체인 구조 도입

- 인증, 헤더 표준화, 관측성 등 필터를 체인으로 분리
- 필터 실행 순서 정의
- 필터 중단 조건 정의

완료 기준:

- 여러 정책 필터를 순서 있게 조합할 수 있다

### 2. 플러그인 계약 정의

- 필터 간 메타데이터 전달 규칙 정의
- 공통 컨텍스트 키 또는 헤더 규칙 정리
- 필터별 입력/출력 기대치 문서화

완료 기준:

- 필터를 추가해도 기존 체인이 깨지지 않는 기본 계약이 생긴다

### 3. 정책 설정 외부화

- route/service/tenant 단위 설정 구조 도입
- 플러그인 enable/disable 구조 추가
- 정책 옵션을 코드가 아닌 설정으로 제어

완료 기준:

- 운영자가 코드 수정 없이 정책 조합을 바꿀 수 있는 기반이 생긴다

### 4. observability 연동

- structured logging
- metrics 추가
- trace metadata 정리
- 필터별 지연 시간 또는 실패율 관찰 포인트 정의

완료 기준:

- 어떤 필터가 요청에 어떤 영향을 줬는지 추적 가능해진다

### 5. 운영 안정성 규칙 정리

- fail-open / fail-close 정책 재정의
- 플러그인 호환성 규칙 정의
- 장애 시 fallback 전략 정의
- 롤백 절차 초안 문서화

완료 기준:

- 정책 배포와 장애 대응을 운영 관점에서 설명할 수 있다

### 6. rate limiting 및 외부 연동 구조 초안

- quota 저장소 연계 방향 정리
- secret/key 연동 구조 정리
- 외부 정책 저장소 또는 control plane 연결 포인트 식별

완료 기준:

- `enterprise` 단계에서 분리될 외부 시스템 경계가 문서와 구조로 드러난다

### 7. 테스트와 검증 체계 확장

- 다중 필터 순서 테스트
- 필터 중단 조건 테스트
- 설정 변경 시 동작 테스트
- 장애 및 fallback 테스트

완료 기준:

- 단일 기능 검증을 넘어 운영 시나리오 기반 테스트가 가능하다

## 최종 완료 기준

- gateway가 단일 데모가 아니라 정책 플러그인 플랫폼 초안으로 보인다
- 여러 정책 필터를 조합하고 관리할 수 있다
- 설정, 관측성, 안정성까지 포함한 운영 관점의 구조가 잡힌다
- 이후 enterprise 단계에서 control plane을 붙일 수 있는 경계가 정리된다
