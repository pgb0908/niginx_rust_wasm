# nginx + Rust WASM API Gateway 아키텍처

## 1. 문서 목적

이 문서는 이 레포가 지향하는 `nginx + Rust WASM` 기반 API gateway의 방향을 정리한다.

이 프로젝트는 처음부터 완성형 제품을 만들기보다, 아래 흐름으로 단계적으로 고도화하는 것을 목표로 한다.

- `v1`: 실행 모델 검증
- `v2`: 공통 정책 계층 도입
- `v3`: 운영 가능한 gateway 플랫폼 초안

레포의 `plan/v1`, `plan/v2`, `plan/v3` 디렉토리는 바로 이 단계별 진화를 담는 공간으로 사용한다.

## 2. 최종 목표

최종 목표는 단순한 reverse proxy가 아니다. `ingress gateway`에서 시작해서, 플러그인을 통해 역할을 점진적으로 확장할 수 있는 gateway 플랫폼을 만드는 것이다.

최종적으로는 아래 성격을 갖는 구조를 목표로 한다.

- 외부 트래픽이 처음 들어오는 ingress gateway
- 공통 인증/인가 정책의 집행 지점
- 조직 공통 헤더/추적 정보의 표준화 지점
- tenant 또는 서비스별 정책의 적용 지점
- 운영 정책을 배포하고 롤백할 수 있는 플러그인 기반 gateway

즉, 이 프로젝트의 핵심은 "nginx 설정을 많이 쓰는 것"이 아니라 "프록시 위에 정책 실행 계층을 안전하게 올리는 것"이다.

## 3. 여기서 말하는 WASM

이 문서에서 말하는 `WASM`은 단순한 바이너리 포맷 이상의 의미를 가진다.

- `WASM` = 컴파일된 모듈 포맷
- `proxy-wasm` = 프록시가 WASM 플러그인을 실행하기 위한 인터페이스/실행 모델

따라서 이 프로젝트에서 `nginx + WASM`은 사실상 아래 의미로 사용한다.

- `nginx`가 WASM을 실행할 수 있는 모듈 또는 런타임과 함께 동작한다
- Rust 코드를 `.wasm`으로 빌드한다
- 이 WASM 모듈은 프록시 요청 처리 중간에서 필터처럼 실행된다

중요한 점은, 순정 `nginx`가 그냥 일반 WASM 파일을 바로 실행하는 것이 아니라는 점이다. 프록시가 요청/응답 정보를 WASM 필터에 넘겨줄 수 있어야 하며, 그 문맥은 보통 `proxy-wasm`으로 이해하는 것이 맞다.

## 4. 왜 Rust를 사용하는가

WASM을 쓴다고 해서 반드시 Rust를 써야 하는 것은 아니다. 하지만 이 프로젝트에서는 Rust가 가장 현실적인 기본 선택이다.

이유는 아래와 같다.

- Rust의 WASM 지원이 가장 성숙한 편이다
- proxy-wasm 관련 예제와 자료가 Rust 쪽에 가장 많이 모여 있다
- 성능과 메모리 모델이 요청 처리 경로에 잘 맞는다
- `nginx`, `Rust`, `WASM`이 모두 낯선 상황에서 언어까지 비주류로 가면 학습 비용이 크게 올라간다

즉, Rust는 필수가 아니라 가장 실용적인 선택이다.

## 5. 전체 아키텍처 개요

이 프로젝트의 기본 구성은 아래와 같다.

```text
Client
  |
  v
Load Balancer / Ingress
  |
  v
nginx gateway
  |
  +--> WASM policy layer
  |
  +--> upstream routing
  |
  +--> backend services

External control plane
  - policy config
  - plugin versioning
  - rollout / rollback
  - tenant rules
  - observability integration
```

구성 요소별 역할은 아래처럼 나눈다.

- `nginx`
  - 프록시 본체
  - 라우팅, upstream 연결, 네트워크 처리 담당

- `WASM policy layer`
  - 요청/응답 경계에서 실행되는 공통 정책 계층
  - 인증, 헤더 처리, 정책 집행, 추적 메타데이터 추가 담당

- `backend services`
  - 실제 비즈니스 로직 담당

- `external control plane`
  - 정책, 설정, 버전, 롤아웃, 운영 통제를 담당

이 구조에서 중요한 원칙은 `데이터 플레인`과 `제어 플레인`을 분리하는 것이다.

## 6. nginx와 WASM의 책임 경계

엔터프라이즈로 갈수록 가장 중요한 것은 책임 경계다.

### nginx에 둘 기능

`nginx`는 프록시 본체 역할에 집중한다.

- TLS 종료
- connection 관리
- keep-alive, timeout, buffering
- upstream 연결과 load balancing
- 경로 기반 라우팅
- retry 및 기본 failover
- 요청 크기 제한
- 압축 및 기본 캐시 전략

즉, 네트워크와 프록시의 핵심 기능은 가능한 한 `nginx`에 둔다.

### WASM에 둘 기능

`WASM`은 공통 정책 계층에 집중한다.

- JWT, API key, tenant 식별 같은 인증 전처리
- 요청/응답 헤더 표준화
- 공통 차단 규칙
- route별 정책 분기
- trace context 전파
- 조직 공통 메타데이터 추가
- 요청 검증과 정책 집행

즉, 여러 서비스 앞단에서 반복적으로 필요한 정책은 WASM에 둔다.

### WASM에 두지 않는 것이 좋은 기능

아래 기능은 WASM에 과도하게 넣지 않는 편이 좋다.

- 복잡한 서비스 디스커버리
- 장기 상태 저장
- 대규모 캐시의 중심 로직
- 긴 외부 DB 트랜잭션
- 운영 설정 관리의 중심
- 복잡한 workflow orchestration

이런 기능은 요청 경로를 무겁게 만들고 운영 복잡도를 크게 높인다.

### 외부 시스템으로 분리할 기능

아래는 control plane 또는 별도 시스템으로 분리하는 것이 맞다.

- 정책 배포
- 플러그인 버전 관리
- tenant별 설정 관리
- secret 및 key rotation 연동
- rate limit quota 저장소
- 중앙 로그/메트릭/트레이싱 수집

정리하면:

- `nginx` = 네트워크와 프록시
- `WASM` = 공통 정책
- 외부 시스템 = 운영 제어

## 7. 다중 WASM 필터 체인

최종적으로는 여러 WASM 필터를 사용할 수 있어야 한다. 다만 이것은 보통 병렬 실행이 아니라 `순차 체인` 구조다.

예를 들면 아래처럼 구성할 수 있다.

- 필터 A: 인증
- 필터 B: tenant 정책
- 필터 C: 헤더 표준화
- 필터 D: 로깅/트레이스 메타데이터 추가

핵심 특성은 아래와 같다.

- 여러 필터를 순서대로 실행할 수 있다
- 실행 순서는 정책 결과에 직접 영향을 준다
- 앞 필터가 요청을 차단하면 뒤 필터는 실행되지 않을 수 있다
- 필터 간에 주고받는 메타데이터 계약이 필요하다

초기에는 필터를 너무 잘게 쪼개지 않는 것이 좋다.

- `v1`: 단일 필터
- `v2`: 인증/관측성 정도로 분리된 소수 필터
- `v3`: 명확한 우선순위와 체인을 가진 플러그인 구조

즉, 최종 목표가 플러그인 플랫폼이더라도 시작은 단순하게 가고, 이후 운영 경험을 바탕으로 체계적으로 분리해야 한다.

## 8. Kong과의 관계

`Kong`도 본질적으로는 플러그인 기반 gateway다. 다만 `Kong`은 `OpenResty/nginx` 위에서 `Lua plugin`을 실행하는 완성형 플랫폼에 가깝다.

개념적으로는 아래처럼 대응된다.

- `Kong plugin` = `WASM policy filter`
- `Kong의 요청 처리 단계` = `proxy-wasm`의 요청/응답 처리 단계
- `Kong의 공통 정책 계층` = `WASM policy layer`

차이는 명확하다.

- `Kong`은 이미 플러그인 플랫폼, 관리 API, 운영 모델이 준비되어 있다
- 이 프로젝트는 `nginx + WASM`으로 그런 구조를 직접 설계하고 키워가는 방향이다

즉, 이 프로젝트는 "Kong 같은 구조를 더 낮은 수준에서 직접 조립하는 학습과 구현"에 가깝다.

## 9. 왜 엔터프라이즈 gateway는 플러그인 시스템으로 가는가

엔터프라이즈 환경에서는 아래 문제가 반복해서 생긴다.

- 서비스 수가 늘어난다
- 팀별 요구사항이 달라진다
- 공통 정책은 계속 늘어난다
- 정책 변경을 서비스 배포와 분리하고 싶어진다

이때 모든 로직을 `nginx.conf`에 쌓거나, 각 서비스 코드 안에 중복 구현하면 운영이 버티지 못한다.

그래서 결국 아래 구조로 수렴한다.

- 고정된 프록시 코어
- 교체 가능하고 조합 가능한 정책 플러그인
- 외부 control plane

플러그인 시스템이 필요한 이유는 아래와 같다.

- 공통 정책을 재사용할 수 있다
- 서비스별로 다른 정책 조합을 적용할 수 있다
- 정책 배포를 서비스 배포와 분리할 수 있다
- 버전, 롤백, 운영 표준화를 할 수 있다

즉, 엔터프라이즈 gateway의 핵심은 단순 프록시 기능보다 `정책을 안전하게 적용하고 운영하는 능력`에 있다.

## 10. 단계별 고도화 방향

이 레포는 `plan/v1`, `plan/v2`, `plan/v3`로 갈수록 기능이 고도화되는 구조로 본다.

- `plan/v1`: 실행 모델 검증 단계
- `plan/v2`: 공통 정책 gateway 단계
- `plan/v3`: 운영 가능한 플러그인 플랫폼 단계

각 단계는 이전 단계를 기반으로 확장되며, 상위 단계로 갈수록 기능을 추가하고 운영 모델을 분리한다.

## 11. plan/v1 기능 목록

`plan/v1`은 "정말 nginx 안에서 Rust WASM이 gateway 필터로 동작하는가"를 검증하는 단계다.

### 목표

- `nginx + Rust WASM` 실행 모델 검증
- 경로 기반 라우팅 검증
- WASM 필터가 요청 처리 경로에서 실제로 실행되는지 검증

### 구현 기능

- `nginx` 기반 ingress gateway 1개
- WASM 실행이 가능한 nginx 런타임 또는 모듈 구성
- Rust 기반 단일 WASM 필터 1개
- `/users/*` 와 `/orders/*` 경로 기반 라우팅
- mock upstream 2개
- `x-gateway-wasm: active` 같은 표시용 헤더 추가
- Docker Compose 기반 실행 환경
- 최소 실행 및 확인 문서

### 이 단계에서 하지 않을 기능

- 인증/인가
- rate limiting
- 외부 설정 저장소
- 플러그인 체인
- observability stack
- 관리 API

### 완료 기준

- `/users/*` 요청이 `users` upstream으로 간다
- `/orders/*` 요청이 `orders` upstream으로 간다
- 외부에서 WASM 필터 동작 여부를 확인할 수 있다

## 12. plan/v2 기능 목록

`plan/v2`는 단순 데모에서 벗어나, 공통 정책 계층으로서의 gateway 가치를 드러내는 단계다.

### 목표

- 여러 서비스 앞단에 공통 정책을 적용할 수 있는 구조 도입
- ingress gateway가 실제 정책 집행 지점으로 동작하도록 확장

### 구현 기능

- 인증 필터 추가
  - JWT 또는 API key 기반 기본 인증
- 공통 request ID / trace header 처리
- 요청/응답 로깅용 메타데이터 추가
- route별 정책 분기
- 표준 에러 응답 형식
- WASM 필터를 2~3개 수준으로 분리하거나, 내부 정책 모듈 구조 분리
- 설정값의 외부화 시작
  - 환경변수 또는 설정 파일 기반
- 기본 테스트 시나리오 정리

### 권장 기능

- tenant 식별 헤더 처리
- 정책 실패 시 기본 동작 정의
  - fail-open 또는 fail-close
- 인증 실패/정책 실패 로그 규칙 정리

### 이 단계에서 아직 미루는 기능

- 중앙 control plane
- 동적 플러그인 배포
- 고급 rate limiting 저장소
- canary rollout

### 완료 기준

- 인증과 공통 정책이 서비스 코드 밖 gateway 계층에서 적용된다
- 서비스별로 다른 정책 조합을 일부 적용할 수 있다
- 단일 필터 데모를 넘어 정책 gateway로서의 가치가 보인다

## 13. plan/v3 기능 목록

`plan/v3`는 운영 가능한 플러그인 플랫폼의 초안을 만드는 단계다.

### 목표

- gateway를 여러 서비스와 팀이 함께 쓸 수 있는 구조로 확장
- 플러그인 기반 정책 운영 모델의 초안 수립

### 구현 기능

- 다중 WASM 필터 체인 구조
- 필터 순서, 우선순위, 중단 조건 정의
- 정책/플러그인 설정의 외부화
- route, service, tenant 단위 정책 적용 구조
- 플러그인 enable/disable 구조
- observability 연동
  - structured logging
  - metrics
  - tracing metadata
- 배포 및 롤백 절차 초안
- 장애 시 fallback 정책 정의
- compatibility 규칙 초안
  - 플러그인 버전과 gateway 런타임 호환성

### 권장 기능

- rate limiting 연동 구조
- secret/key 연동 구조
- 정책 검증용 사전 체크
- 테스트 자동화 확대

### 이 단계의 핵심 산출물

- "필터 하나를 실행하는 데모"가 아니라
- "플러그인 정책을 운영 가능한 형태로 조합하는 gateway 초안"

### 완료 기준

- 여러 정책 필터를 순서 있게 조합할 수 있다
- 설정과 정책이 코드에 하드코딩되지 않는다
- 운영자가 정책을 바꾸고 되돌리는 흐름을 설계할 수 있다

## 14. 엔터프라이즈 단계에서 추가될 것

`v3` 이후의 엔터프라이즈 단계에서는 control plane과 운영 표준화가 중심이 된다.

추가로 필요한 요소는 아래와 같다.

- policy versioning
- canary rollout
- plugin compatibility 관리
- 중앙 observability
- tenant별 정책 관리
- secret lifecycle 연동
- 운영용 관리 API 또는 제어 인터페이스

이 단계에서는 "필터를 잘 만들었는가"보다 "정책을 안전하게 배포하고 회수할 수 있는가"가 더 중요해진다.

## 15. 설계 원칙

이 프로젝트를 끝까지 가져가려면 아래 원칙을 유지해야 한다.

- WASM은 가능한 한 짧고 빠르게 끝나야 한다
- WASM은 가능하면 stateless에 가깝게 유지한다
- `nginx`에는 네트워크와 프록시 기능을 남긴다
- 공통 정책은 WASM으로 올린다
- 운영 제어는 외부 시스템으로 분리한다
- 다중 필터를 사용할 경우 순서와 우선순위를 명확히 관리한다
- 필터를 나눌 때는 기능 분리보다 운영 가능성을 우선한다
- 설정은 코드에 하드코딩하지 않고 외부에서 주입한다
- 정책 실패 시 동작을 명확히 정의한다

## 16. 정리

이 레포의 방향은 아래 한 줄로 요약할 수 있다.

`ingress gateway`에서 시작해, `nginx`를 프록시 코어로 두고, `Rust WASM`을 정책 플러그인 계층으로 사용하며, `plan/v1 -> plan/v3`로 갈수록 운영 가능한 gateway 플랫폼으로 고도화한다.

따라서 지금 해야 할 일은 단순히 필터 하나를 붙이는 것이 아니라, 이후 플러그인 플랫폼으로 자연스럽게 확장될 수 있는 책임 경계와 단계별 기능을 유지하면서 구현을 쌓아가는 것이다.
