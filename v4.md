# v4 개발 명세

## 1. 문서 목적

이 문서는 이 레포의 `v4`를 "실무에서 바로 사용할 수 있는 구조"로 끌어올리기 위한 구현 명세다.

`v1`은 실행 모델 검증, `v2`는 공통 정책 도입, `v3`는 운영 가능한 플러그인 플랫폼 초안이었다.  
`v4`는 초안 수준을 끝내고 아래를 실제 구현 대상으로 삼는다.

- 데이터 플레인과 제어 플레인의 명확한 분리
- 정책 배포, 검증, 롤백의 자동화
- 관측성의 기본 완성
- secret, rate limit, tenant 정책의 외부 시스템 연동
- 운영자와 서비스 팀이 함께 쓸 수 있는 제품형 구조

이 문서는 설명용 개요가 아니라 구현 지시서다.  
개발 에이전트는 이 문서만 읽고 디렉터리 생성, 서비스 구현, API 정의, 테스트 작성까지 바로 시작할 수 있어야 한다.

## 2. v4 목표

`v4`의 목표는 다음과 같다.

- gateway를 단일 데모가 아니라 운영 가능한 제품형 시스템으로 만든다
- 정책 변경을 서비스 배포와 분리한다
- route, service, tenant 단위 정책을 중앙에서 관리한다
- Wasm 플러그인을 버전 단위로 배포하고 롤백할 수 있다
- 장애 시 fallback 규칙과 운영 절차가 코드와 문서에 같이 존재한다
- 로그, 메트릭, 트레이싱을 통해 요청 단위로 문제를 추적할 수 있다

## 3. v4 비목표

이번 단계에서 하지 않는 것:

- 멀티 리전 글로벌 트래픽 분산
- 완성형 웹 관리자 UI
- 대규모 분산 quota 엔진의 고급 최적화
- 조직 전체 IAM 제품화

즉, `v4`는 "프로덕션 사용 가능"을 목표로 하지만, 대형 SaaS 플랫폼의 최종 형태까지 한 번에 가는 것은 아니다.

## 4. 최종 산출물

`v4`에서 반드시 남아야 하는 산출물은 아래와 같다.

- `v4/` 데이터 플레인 예제 구현
- `control-plane/` 정책 관리 API 및 배포 엔진
- `configs/` 정책 스키마와 예시 데이터
- `docs/` 운영 절차, 장애 대응, 호환성 규칙 문서
- `tests/` 단위, 통합, 회귀, 장애, 롤백 테스트
- `scripts/` 배포, 검증, 롤백 자동화 스크립트

## 5. 목표 아키텍처

```text
Client
  |
  v
Load Balancer
  |
  v
nginx + wasm data plane
  |
  +--> auth/filter chain
  +--> route decision
  +--> rate limit check
  +--> upstream services
  |
  +--> metrics/logs/traces

Control plane
  |
  +--> policy API
  +--> plugin registry
  +--> rollout manager
  +--> config validator
  +--> secret reference resolver
  +--> audit log

External systems
  |
  +--> Redis for rate limit/quota
  +--> PostgreSQL for policy metadata/audit
  +--> Prometheus
  +--> OpenTelemetry collector
  +--> Vault or compatible secret store
```

## 6. 핵심 원칙

모든 구현은 아래 원칙을 따라야 한다.

- nginx는 네트워크와 프록시 코어에 집중한다
- Wasm은 공통 정책 집행에 집중한다
- 운영 설정은 control plane이 관리한다
- 요청 경로는 stateless에 가깝게 유지한다
- 정책 배포는 서비스 배포와 분리한다
- 모든 변경은 검증 가능하고 되돌릴 수 있어야 한다
- 정책 결정과 실패 원인은 로그와 메트릭에 남아야 한다

## 7. 필수 기능

### 7.1 Control Plane

반드시 구현할 기능:

- 정책 CRUD API
- plugin registry API
- route, service, tenant policy API
- rollout 생성, 승인, 배포, 취소 API
- config validation API
- compatibility check API
- audit log 조회 API

최소 요구사항:

- HTTP JSON API로 제공
- OpenAPI 문서 생성 가능 구조
- PostgreSQL 사용
- 모든 변경은 revision 번호를 가진다
- 배포는 immutable revision 단위로 수행한다

### 7.2 Data Plane

반드시 구현할 기능:

- nginx 기반 ingress
- 다중 Wasm 체인 실행
- control plane이 생성한 정적 배포 산출물 로드
- route/service/tenant 기반 정책 적용
- 요청 차단, 허용, 헤더 표준화, trace propagation
- rate limit 조회
- fallback 정책 수행

최소 요구사항:

- 설정 파일과 Wasm 모듈을 revision 단위 디렉터리로 로드
- reload 전 `nginx -t` 검증
- 실패 시 이전 revision 유지

### 7.3 Plugin Runtime Contract

반드시 정의하고 구현해야 할 계약:

- 플러그인 입력 헤더
- 플러그인 출력 헤더
- 공통 메타데이터 키
- 에러 반환 포맷
- plugin lifecycle 훅
- fail-open / fail-close / fail-static-response
- plugin version compatibility

### 7.4 Observability

반드시 구현할 기능:

- JSON structured access log
- plugin별 decision log
- Prometheus metrics endpoint
- OpenTelemetry trace context propagation
- request_id, trace_id, tenant_id, route_id, plugin_chain, revision 노출

최소 메트릭:

- `gateway_requests_total`
- `gateway_request_duration_ms`
- `gateway_plugin_executions_total`
- `gateway_plugin_failures_total`
- `gateway_policy_denied_total`
- `gateway_rate_limit_denied_total`
- `gateway_reload_total`
- `gateway_reload_failures_total`

### 7.5 Rollout and Rollback

반드시 구현할 기능:

- 새 revision 생성
- canary rollout
- staged rollout
- immediate rollback
- health signal 기반 자동 중단

최소 요구사항:

- rollout 상태 저장
- revision별 plugin, config, schema hash 저장
- rollback 시 이전 revision으로 atomic 전환

### 7.6 Secret and External Integration

반드시 구현할 기능:

- secret reference만 정책에 저장
- 실제 secret 값은 Vault 또는 환경 대체 저장소에서 resolve
- rate limit 저장소는 Redis 사용
- tracing/export는 OpenTelemetry collector 사용

## 9. 구현 대상 서비스

### 9.1 Control Plane Service

추천 구현:

- Rust `axum` 또는 Python `FastAPI`
- PostgreSQL
- SQL migration 도구 포함

서비스 책임:

- 정책 저장
- revision 생성
- rollout orchestration
- 호환성 검증
- 감사 로그 저장

### 9.2 Revision Builder

별도 모듈 또는 control plane 내부 모듈로 구현한다.

책임:

- 정책 revision 입력을 받아 최종 data plane 산출물 생성
- nginx 템플릿 렌더링
- plugin manifest 생성
- config bundle 압축
- schema validation 수행
- hash 계산

출력:

- `bundle.tar.gz`
- `manifest.json`
- `checksums.txt`

### 9.3 Agent

데이터 플레인 노드에 배포되는 경량 에이전트를 둔다.

책임:

- control plane에서 승인된 revision polling
- bundle download
- checksum 검증
- revision 디렉터리 배치
- `nginx -t` 실행
- 성공 시 symlink 전환 및 reload
- 실패 시 rollback
- 상태 보고

중요:

- v4에서 말하는 "agent"는 LLM 에이전트가 아니라 node side deploy agent다.

## 10. 필수 플러그인 목록

`v4/plugins/`에는 최소 아래 플러그인을 구현한다.

### 10.1 auth-filter

책임:

- API key 또는 JWT 검증
- secret reference 기반 키 조회 결과 사용
- 인증 실패 시 표준 JSON 응답 반환

입력:

- `Authorization`
- `x-api-key`
- `x-tenant-id`

출력:

- `x-auth-subject`
- `x-auth-method`
- `x-gateway-decision`

실패 정책:

- 기본 `fail-close`

### 10.2 tenant-filter

책임:

- tenant 식별
- tenant별 plugin enable/disable 적용
- tenant별 정책 프로필 선택

실패 정책:

- 기본 `fail-close`

### 10.3 header-filter

책임:

- 공통 헤더 주입
- request_id, trace_id, organization, route_id 표준화

실패 정책:

- 기본 `fail-open`

### 10.4 rate-limit-filter

책임:

- Redis 기반 rate limit 조회 및 차감
- tenant/service/route 기준 quota 적용

실패 정책:

- 정책별 선택 가능
- 기본은 `fail-open` 이 아니라 `configurable`

주의:

- 금융성 또는 민감 API는 `fail-close`
- 공개성 읽기 API는 `fail-open` 허용 가능

### 10.5 observe-filter

책임:

- trace metadata 보강
- plugin execution 결과 헤더화
- timing metadata 기록

실패 정책:

- 기본 `fail-open`

## 11. 플러그인 계약

모든 플러그인은 아래 계약을 따라야 한다.

### 11.1 공통 헤더

- `x-request-id`
- `x-trace-id`
- `x-tenant-id`
- `x-route-id`
- `x-service-id`
- `x-gateway-revision`
- `x-gateway-plugin-chain`
- `x-gateway-decision`
- `x-gateway-decision-reason`
- `x-gateway-policy-profile`
- `x-gateway-plugin-version`

### 11.2 응답 에러 형식

모든 차단 응답은 아래 형식을 따른다.

```json
{
  "error": {
    "code": "unauthorized",
    "message": "missing or invalid api key",
    "request_id": "req-123",
    "trace_id": "trace-123",
    "route_id": "users-read"
  }
}
```

### 11.3 메타데이터 전달 규칙

- 기존 헤더를 삭제하지 않는다
- 새 헤더는 prefix 규칙을 따른다
- 정책 결정 결과는 `x-gateway-decision`에 누적 가능한 문자열이 아니라 최종 상태만 기록한다
- 세부 원인은 `x-gateway-decision-reason` 또는 로그 필드로 남긴다

## 14. Revision 배포 구조

데이터 플레인은 revision 단위 디렉터리를 읽어야 한다.

예시:

```text
/opt/gateway/runtime-config/revisions/20260423-001/
  manifest.json
  nginx.conf
  plugins/
    auth_filter.wasm
    tenant_filter.wasm
    header_filter.wasm
    rate_limit_filter.wasm
    observe_filter.wasm
  policies/
    routes.json
    tenants.json
    services.json
    rollout.json
```

현재 활성 revision은 symlink로 가리킨다.

```text
/opt/gateway/runtime-config/current -> revisions/20260423-001
```

전환 절차:

1. 새 revision bundle download
2. checksum 검증
3. revisions 디렉터리에 압축 해제
4. `nginx -t` 검증
5. 성공 시 `current` symlink 교체
6. `nginx -s reload`
7. health check 및 메트릭 확인
8. 실패 시 이전 symlink 복원

## 15. Rollout 정책

`v4`는 최소 두 가지 rollout을 지원해야 한다.

### 15.1 Full Rollout

- 모든 gateway 인스턴스에 한 번에 배포
- 개발/스테이징 환경에서 기본 사용

### 15.2 Canary Rollout

- 일부 인스턴스 또는 일부 라우트에 우선 적용
- 운영 환경 기본 방식

canary 중단 조건:

- `5xx` 비율 증가
- 인증 실패율 비정상 상승
- plugin failure 증가
- p95 latency 증가

자동 중단 시 동작:

- rollout 상태를 `paused` 또는 `failed`로 변경
- 이전 stable revision 유지
- 운영자 알림 발생

## 16. Secret 관리

정책 파일에 secret 값 자체를 두면 안 된다.

허용:

```yaml
auth:
  api_key_secret_ref: vault://gateway/users-api-key
```

금지:

```yaml
auth:
  api_key: users-secret
```

구현 요구사항:

- local dev에서는 `.env` 또는 dev secret file fallback 허용
- 운영 환경에서는 Vault 또는 호환 저장소를 통해 resolve
- secret 조회 실패 시 plugin fail mode에 따라 처리

## 17. Rate Limiting

`v4`는 초안이 아니라 실제 동작하는 rate limiting을 가져야 한다.

요구사항:

- Redis 사용
- key 구성은 `tenant + route + period`
- burst와 steady rate 구분 가능
- deny 시 표준 JSON 응답 반환
- 응답 헤더에 quota 상태 노출

예시 헤더:

- `x-ratelimit-limit`
- `x-ratelimit-remaining`
- `x-ratelimit-reset`

## 19. 장애 대응 규칙

각 플러그인과 외부 연동은 아래 규칙을 명시해야 한다.

- Redis 장애 시 rate limit filter 동작
- Vault 장애 시 auth filter 동작
- Control plane 장애 시 기존 deployed revision 유지
- plugin load 실패 시 gateway startup 또는 reload 실패
- partial rollout 실패 시 canary 중단 및 stable 유지

운영 문서에 반드시 포함할 runbook:

- 인증 실패 급증
- reload 실패
- 특정 revision 배포 실패
- Redis 연결 장애
- Vault 연결 장애
- OpenTelemetry collector 장애

## 23. 완료 기준

`v4`는 아래 조건을 모두 만족해야 완료로 본다.

- 새 정책을 control plane API로 생성할 수 있다
- revision 생성과 validation이 자동 수행된다
- agent가 새 revision을 받아 검증 후 gateway에 배포한다
- 실패 시 자동 rollback 된다
- Redis 기반 rate limiting이 실제 동작한다
- secret ref 기반 인증이 실제 동작한다
- access log, metrics, traces로 요청 추적이 가능하다
- canary rollout과 rollback이 테스트로 검증된다
- 운영 문서와 runbook이 포함된다

## 24. 우선 구현 제안

에이전트는 처음부터 모든 기능을 한 번에 완성하려 하지 말고, 아래 3개를 우선 완성한다.

1. `control plane + revision builder`
2. `agent + atomic deploy/rollback`
3. `auth/rate-limit/observe plugin + observability`

이 세 축이 완성되어야 `v4`가 실무 투입 가능한 최소 수준이 된다.
